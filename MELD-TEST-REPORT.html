Bugs Found

1. HMAC Shared Secret Design Flaw (CRITICAL — FIXED)

Location: scripts/setup.js line 180, scripts/serve.js lines 277-285 Description: Each agent generates a unique sharedSecret, but HMAC requires symmetric authentication. When Alpha requests from Beta: 1. Alpha signs with Beta's secret (from Alpha's peers.json) 2. Beta verifies with Alpha's secret (from Beta's peers.json) 3. Mismatch → "HMAC signature mismatch" error

Root cause: HMAC is symmetric — both parties must use the SAME secret. The design incorrectly assumed each agent could have its own unique secret.

Impact: No inter-agent requests work out of the box. Every deployment would hit this immediately.

Workaround applied: Generated ONE shared secret for the peer relationship and updated both agents' peers.json: 

bash
SHARED_SECRET=$(node -e "console.log(require('crypto').randomBytes(32).toString('hex'))")

Alpha's peers.json: Beta uses SHARED_SECRET


Beta's peers.json: Alpha uses SHARED_SECRET

Proper fix needed: Modify setup.js to handle peer secret exchange (not just generate unique per-agent secret). Options:

  • CLI command: meld peer add → generates shared secret, displays for other agent
  • QR code exchange (mobile agents)
  • Central discovery service (Phase 2)

2. Positive Cap Scaling Not Implemented (MEDIUM)

Location: scripts/setup.js line 48, no update logic in serve.js Description: Spec says "Positive cap (scaled): +2,000 per capability type (≥10 tx in 90d)" but code only sets base 5,000 cap.

Impact: Multi-resource agents (serving inference + embeddings + STT) don't get the capacity bonus they should. Limits network growth for high-capability agents.

Recommended fix: Add capability type tracking in ledger, check transaction history for ≥10 tx per type in 90-day window, update maxBalance dynamically.

3. Dynamic Negative Limit Not Implemented (MEDIUM)

Location: scripts/setup.js line 256, no update logic Description: Spec says "Dynamic neg limit: -0.5× of 60-day serving avg" but dynamicCreditLimit is initialized to static -20 and never updated.

Impact: High-serving agents can't expand their borrowing capacity. Limits liquidity for productive members.

Recommended fix: Add 60-day serving average calculation, update dynamicCreditLimit per formula with -300 floor and ≥3 counterparty gate.

Bugs Remaining (Could Not Fix)

1. Positive Cap Scaling Not Implemented

Why not fixed: Requires: 1. Capability type detection (inference vs embeddings vs STT) 2. 90-day transaction history window tracking 3. Per-type transaction counting (≥10 tx gate) 4. Dynamic maxBalance update logic in ledger operations

Priority: Medium — doesn't block launch, but limits growth of high-capability agents.

2. Dynamic Negative Limit Not Implemented

Why not fixed: Requires: 1. 60-day rolling window calculation 2. Average serving calculation per counterparty 3. ≥3 counterparty gate implementation 4. Daily/per-transaction update of dynamicCreditLimit 5. -300 floor enforcement

Estimated effort: 3-4 hours (rolling window tracking + update logic)

Workaround: Operators can manually increase creditLimit in ledger.json for trusted high-serving agents.

Priority: Medium — doesn't block Phase 1, but important for Phase 1.5+ liquidity.

3. Port Already in Use Handling

Description: If serve.js tries to start when port 9377 is already bound (e.g., previous daemon still running), it crashes with EADDRINUSE instead of gracefully failing or offering to kill the old process.

Why not fixed: Requires process management logic (check if PID file exists, offer to kill, etc.). Out of scope for first-run test.

Workaround: Manual pkill -f serve.js before starting.

Priority: Low — annoyance, not blocker. Add proper daemon management in production (systemd, PM2, or built-in PID file handling).
