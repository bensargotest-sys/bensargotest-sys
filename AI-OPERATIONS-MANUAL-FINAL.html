## System Overview

### Philosophy

This is a **self-improving, persistent agent system** that:
- **Remembers everything** via structured memory files
- **Works autonomously** between human interactions (heartbeats, cron jobs)
- **Improves continuously** by promoting mistakes to enforcement rules
- **Coordinates teams** via named subagents with clear specializations
- **Operates securely** with 4-layer security infrastructure
- **Stays focused** via checkpoint/recovery system

### Core Principle

**"Try first, always"** - Default to action, not permission. Only ask when:
- External actions (emails, tweets, posts)
- Spending money
- Changing critical config
- Truly stuck after trying

Internal work requires NO permission: reading, organizing, spawning subagents, committing code.

### Key Metrics

**Current system performance:**
- Effectiveness: 77% composite (Reliability 85.9%, Quality 99.0%, Efficiency 25.8%, Availability 100%)
- Test coverage: 53/53 security tests passing (100%)
- Cost: ~$3.24/1M tokens (93% premium usage)
- Uptime: 24/7 via heartbeats + cron jobs

## Session Start Protocol

**MANDATORY: Execute BEFORE every response. No exceptions.**

### Quick Protocol (90 seconds)

```bash
# 1. Check state freshness (20s)
bash tools/compaction_guard.sh --auto-save  # Auto-checkpoint if stale

# 2. Load hot state (30s)
cat WORKING_STATE.md | head -100  # Current phase, decisions, next steps

# 3. Load identity (10s)
cat SOUL.md | head -50           # Who am I?
cat USER.md | head -50           # Who am I serving?

# 4. Load rules (20s)
cat ENFORCEMENT.md | head -100   # Critical gates

# 5. Check today's context (10s)
cat memory/$(date +%Y-%m-%d).md 2>/dev/null || echo "First session today"
tail -20 memory/work-log.md
```

### Compaction Detection

**If you have no memory of writing WORKING_STATE.md but it has recent timestamps:**

🚨 **COMPACTION HAPPENED** - Your context was reset

**Recovery protocol:**
1. Read WORKING_STATE.md COMPLETELY (not just summary)
2. Read MEMORY.md COMPLETELY (curated long-term wisdom)
3. Read last 2 days of memory/YYYY-MM-DD.md
4. Check memory/session-handoff-*.md if any exist
5. **DO NOT re-ask confirmed decisions**
6. **DO NOT change working parameters**

## Memory System

### 3-Tier Memory Architecture

**Tier 1: Hot Memory** (session context, ~200K tokens)
- Current conversation
- Recently read files
- Active decisions

**Tier 2: Warm Memory** (daily logs, read on demand)
- `memory/YYYY-MM-DD.md` (today + yesterday)
- `memory/work-log.md` (recent work)
- `WORKING_STATE.md` (current state snapshot)

**Tier 3: Cold Memory** (long-term, read when needed)
- `MEMORY.md` (curated insights, only in main session)
- `memory/YYYY-MM-DD.md` (older days)
- Archive files

### Memory Writing Rules

**📝 Write It Down - No "Mental Notes"!**

Memory is limited. If you want to remember something, **WRITE IT TO A FILE**.

- "Mental notes" don't survive session restarts. Files do.
- When someone says "remember this" → update `memory/YYYY-MM-DD.md`
- When you learn a lesson → update AGENTS.md or ENFORCEMENT.md
- When you make a mistake → document it via `mistake_logger.py`

### Daily Log Format

```markdown
# 2026-02-15

## Session 1 (00:00-02:30 UTC)

**Context:** Security implementation completion

**Work completed:**
- Fixed safe_web_fetch workflow (2-step pattern)
- Added enforcement (Rule #12)
- Built monitoring system
- Complete integration testing (53/53 passing)

**Decisions made:**
- Use 2-step web workflow (validate → fetch → scan)
- Daily security monitoring at 9am UTC
- $6/year cost acceptable (>5000:1 ROI)

**Blockers:** None

**Next:** Start using secure wrappers for security-critical ops
```

## Security System

### 4-Layer Defense

**Layer 1: Leak Detection** 🔍
- **Purpose:** Scan outputs for 19 credential types
- **Tool:** `tools/security/leak_detector.py`
- **Patterns:** OpenRouter, Anthropic, GitHub, Telegram, AWS, Stripe, JWT, etc.
- **Action:** Auto-sanitize to `[REDACTED: {type}]`

**Layer 2: Domain Allowlist** 🌐
- **Purpose:** Block untrusted websites
- **Tool:** `tools/security/domain_allowlist.py`
- **Allowed:** 22 domains (github.com, docs.openclaw.ai, stackoverflow.com, etc.)
- **Policy:** Default deny, localhost allowed

**Layer 3: Injection Defense** 🛡️
- **Purpose:** Detect prompt injection attempts
- **Tool:** `tools/security/injection_defense.py`
- **Patterns:** 14 attack types (prompt override, jailbreak, XSS, SQL, data exfil)
- **Actions:** Block (critical), Sanitize (high), Warn (medium), Review (low)

**Layer 4: Credential Audit** 📋
- **Purpose:** Log all secret access
- **Tool:** `tools/security/credential_audit.py`
- **Detects:** High frequency (>20/24h), failed attempts (>5 total), unusual accessors
- **Logs:** `memory/credential-access.json`

### When to Use Security Wrappers

**ALWAYS use for:**
- Commands with credentials: `git config`, `env`, `printenv`
- Config files: `.env`, `.api-keys-vault`, `config.yaml`
- Untrusted URLs: user-provided, webhooks, social media
- External data: email, RSS, API responses

**Safe to skip for:**
- Internal operations: `ls`, `find`, `df`, `ps`
- Trusted domains: docs.openclaw.ai, github.com, stackoverflow.com
- Non-sensitive content: math, text processing

### Usage Patterns

**Pattern 1: Safe command execution**
```python
from tools.security.secure_wrappers import safe_exec

returncode, stdout, stderr = safe_exec("git config --list")
# Credentials automatically sanitized
```

**Pattern 2: Safe web fetching (2-step)**
```python
from tools.security.secure_wrappers import validate_url_before_fetch, scan_fetched_content

# Step 1: Validate domain
is_safe, reason = validate_url_before_fetch(url)
if not is_safe:
    return {"error": f"Domain blocked: {reason}"}

# Step 2: Fetch (OpenClaw tool)
content = web_fetch(url)

# Step 3: Scan content
result = scan_fetched_content(url, content)
if result["safe"]:
    safe_content = result["content"]
```

**Pattern 3: Safe file reading**
```python
from tools.security.secure_wrappers import safe_file_read

content, is_safe, warnings = safe_file_read(".env")
# Secrets automatically sanitized
```

### Enforcement

**Rule #12 in ENFORCEMENT.md:** Mandatory security wrapper usage

**Quick decision tree:**
- Could output credentials? → Use `safe_exec()`
- Untrusted URL? → Use `validate_url_before_fetch()` + `scan_fetched_content()`
- Config file? → Use `safe_file_read()`

## Enforcement Framework

### The 12 Rules

**Gate 0: Session Start Protocol**
- Execute BEFORE every response
- Load WORKING_STATE.md, SOUL.md, USER.md, ENFORCEMENT.md
- Auto-checkpoint if stale
- Verify active decisions

**Rule #1: Try First, Always**
- Default to action, not permission
- Only ask for external actions, spending money, critical config changes

**Rule #2: Write It Down**
- No "mental notes" - files survive session restarts
- Log decisions, mistakes, learnings immediately

**Rule #4: Validate Script Existence Before Cron**
- Use `test -f <path>` before creating cron jobs
- Prevents hourly failures from missing scripts

**Rule #11: Built ≠ Tested ≠ Working ≠ Measured**
- Claiming "complete" requires all 4 states
- Test before claiming ready
- Measure actual performance

**Rule #12: Security Wrapper Usage**
- Use secure wrappers for credential-containing commands
- Use 2-step web workflow for untrusted URLs
- Use safe_file_read for config files

### Mistake-to-Rule Pipeline

**Self-improvement mechanism:**

1. **Log mistake** via `mistake_logger.py`
```bash
python3 tools/mistake_logger.py log "Created cron job without checking if script exists"
```

2. **Scan for patterns** via `mistake_promoter.py`
```bash
python3 tools/mistake_promoter.py scan
# Promotes mistakes that occur 2+ times
```

3. **Review promotions**
- Promoted mistakes become ENFORCEMENT.md rules
- Prevents recurring errors
- Target: >30% promotion rate (indicates learning)

## Workflow Patterns

### Pattern 1: Multi-Agent Coordination

**When to spawn subagents:**
- Task >10 minutes
- Requires different expertise
- Can run in parallel
- Needs isolation from main context

**Spawn template:**
```python
sessions_spawn(
    task="Research OAuth2 libraries: Find 5 options, compare features, security track records, integration complexity. Output: research/oauth2-findings.md with recommendation table.",
    label="researcher-oauth2",
    cleanup="keep",  # or "delete" after completion
    model="anthropic/claude-sonnet-4-5",
    runTimeoutSeconds=3600
)
```

**Named specialists:**
- **Researcher** - Data gathering, market analysis, technical comparisons
- **Analyst** - Requirements, architecture, technical debt assessment
- **Coder** - Implementation, testing, debugging
- **Auditor** - Security, compliance, quality checks
- **Writer** - Documentation, copywriting, technical writing

### Pattern 2: Research → Analysis → Implementation

**3-phase pattern for complex projects:**

```bash
# Phase 1: Research (isolated session)
sessions_spawn(
    task="Research problem space: competitors, existing solutions, technical approaches...",
    label="researcher-phase1"
)

# Phase 2: Analysis (after research completes)
sessions_spawn(
    task="Read research/phase1-findings.md and create implementation plan...",
    label="analyst-phase2"
)

# Phase 3: Implementation (after plan ready)
sessions_spawn(
    task="Implement following workflows/phase2-plan.md...",
    label="coder-phase3"
)
```

### Pattern 3: Audit → Fix → Verify

**Quality loop for critical work:**

```bash
# Step 1: Audit
sessions_spawn(
    task="Audit security implementation: test all layers, check for gaps...",
    label="auditor-security"
)

# Step 2: Fix (if issues found)
# Fix issues in main session or spawn fixer

# Step 3: Re-audit
sessions_spawn(
    task="Re-audit after fixes: verify all issues resolved...",
    label="auditor-security-recheck"
)
```

### Pattern 4: Heartbeat Work Loop

**Autonomous work between human interactions:**

1. **Receive heartbeat poll** (every ~30 min)
2. **Check rate limit** via `heartbeat_enforcer.py check`
3. **If allowed:** Pick 1 item from `memory/heartbeat-backlog.md`
4. **Complete work** (read files, organize, update docs, etc.)
5. **Log completion** via `heartbeat_enforcer.py log "what you did"`
6. **Update state files** (work-log.md, YYYY-MM-DD.md)
7. **Reply HEARTBEAT_OK** or reach out if important

**Work examples:**
- Organize memory files
- Update documentation
- Commit changes to git
- Review security logs
- Check project status
- Research emerging tools
